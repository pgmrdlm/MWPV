MWPV – NEXT PHASE PLAN (Security)

============================================================
A) PASSWORD STRENGTH METER (small, self-contained)
------------------------------------------------------------
Goal:
  - Ship a warning-only meter so it stops getting postponed.

Scope:
  - Live meter in PasswordEntryWindow showing: Weak / Fair / Good / Strong
  - Policy hint: 8+ chars AND at least 2 of 3 (upper, lower, special)
  - No dictionary checks. No submit blocking.
  - Wipe temp char[] after scoring; still call UICleaner.Clear(...) on submit/cancel

Definition of Done:
  - Meter updates as user types
  - Submit flow unchanged (policy enforced at submit as today)
  - Security_Documentation.md updated from “planned” -> “implemented”


============================================================
B) LOGGING FOUNDATION (data model + API)
------------------------------------------------------------
Goal:
  - Capture events now; add details later.

Schema (encrypted via SQLCipher):
  - LogEvents(EventId, Utc, EventType, Entity, EntityId, UserTag, SessionId, Source, Note)
  - LogEventDetails(DetailId, EventId FK, ColumnName, ChangeType, BeforeHash?, AfterHash?, BeforeEnc?, AfterEnc?, MaskHint?)
  - Indexes: Utc; Entity+EntityId; Details.EventId

Service:
  - ISecureLogger.LogAsync(AuditEvent evt, IEnumerable<ColumnChange> details)
  - ISecureLogger.LogSimpleAsync(AuditEvent evt, params string[] changedColumns)  (MVP)
  - Uses LogPayloadKey from SEDS; AES-GCM helpers exist for future encrypted values

Ingestion Hook:
  - After successful login: ingest DPAPI .elogp files -> write summary event (IngestedEarlyFailure)
  - Show non-blocking banner; securely delete .elogp after ingest

Definition of Done:
  - Tables created/migrated
  - One real code path logs Create/Update with column NAMES only (no values)
  - Ingestion path writes a summary event

C) LOG VIEWER MVP (usable, not fancy)
------------------------------------------------------------
Placement:
  - Viewer takes over the main panel (replacing Category/Detail + Add buttons)
  - Toggle via Tools -> “View Logs…” (clean up menu labels/order)

UI:
  - Top filters: Date range | Event Type | Entity | Search
  - Grid: Time | Type | Entity | EntityId | ColumnsChanged | Note
  - Right panel: list of changed column names (no values for MVP)
  - Session banner: “Early failures ingested: N” (if any)

Definition of Done:
  - Filters work; grid shows LogEvents
  - Selecting a row shows its column list (from LogEventDetails)
  - Handles empty states and stays responsive on real data


============================================================
D) COLUMN-LEVEL ENCRYPTION (design first, implement selectively)
------------------------------------------------------------
Policy:
  - Sensitive user-entered columns: AES‑256‑GCM per column (confidentiality + integrity)
  - Key: separate, randomly generated UserSecretsKey (distinct from DbPassword / LogPayloadKey)
  - Key already stored in encrypted keyfile JSON

Plan:
  - Start logging only column NAMES (and optional hashes)
  - When needed, add per-column encrypted before/after payloads (nonce|ct|tag)
  - UI shows “(encrypted)” or redacted for those fields

Definition of Done (later):
  - At least one sensitive field wired end-to-end as reference
